Partner auth

Hosted identity login for partner products.

Login with OTRUST lets apps send users through a secure, branded identity-proof flow. Partners keep continuity in the user journey while OTRUST keeps the disclosure, DPO contact, and proof verification layer visible.

How it works The real login screen opens from an app-generated challenge.
1
Create or reuse a proof

The user creates an OTRUST identity proof on the themed flow, or reuses an existing proof they control.

2
Open an app challenge

The app creates a short-lived challenge and redirects the user to the OTRUST hosted login screen.

3
Verify and return

OTRUST verifies the proof and redirects back with a token the partner application can validate.

What this gives partners and users

A login flow where identity checks are explicit, scoped, reusable, and visually connected to the partner product.

Passwordless access

Users authenticate by proving control of their OTRUST proof instead of creating another password.

Scoped verification

Apps ask for scopes such as identity verification. The user only presents what the app requested.

Less data exposure

Verification happens from proof metadata. Apps do not need raw document scans or selfie data.

Partner continuity

Approved logo, color, headline, and footer copy follow the hosted login and proof creation screens.

Developer integration

Add a Login with OTRUST button by creating a challenge server-side, redirecting to the returned login URL, then verifying the callback token.

Create a challenge

import { auth } from '@otrust/sdk'

const state = auth.generateState()
const challenge = await auth.createChallenge({
  clientId: 'my-app',
  redirectUri: 'https://app.example/auth/callback',
  scope: ['identity'],
  state
})

if (challenge.ok) {
  res.redirect(challenge.value.loginUrl)
}

Verify the callback

const token = req.query.token
const result = await auth.verify(token)

if (result.ok && result.value.valid) {
  const proofId = result.value.proofId
  const verified = result.value.identity?.verified
}
Direct visits to /auth/login are intentionally not useful. That URL needs a short-lived challenge from your application.

Partner branding

Hosted login and proof creation can inherit approved partner colors and copy per clientId while keeping OTRUST disclosure, the OTRUST footer mark, and DPO contact visible.

  "autoRedirectSeconds": 3
}

Brand-guide checklist

  • Logo must be served over HTTPS and stay below 200KB.
  • Colors are validated as hex, rgb, or rgba values.
  • Copy is rendered as text only, never as injected HTML.
  • The same theme follows users to /proof?auth_challenge=... if they need to create a proof first.
  • Every change is written to the OTRUST audit log with client_id, actor, fields, and timestamp.
