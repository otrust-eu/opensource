  • Independent verification: records can be checked without trusting private backend state alone.

SDK

Integrate OTRUST into your applications with official SDKs.

JavaScript / TypeScript

npm install @otrust/sdk

Quick Example

// Timestamp a file
import { timestamp, hashFile } from '@otrust/sdk'

const hash = await hashFile(file)
const result = await timestamp.create(hash)
console.log(result.id) // ot_xxx...

// Verify a timestamp
const verification = await timestamp.verify(hash)
console.log(verification.status) // 'confirmed'

Timestamp

Verify when data existed using Bitcoin blockchain anchoring.

Timestamp flow

  1. Your file is hashed locally (SHA-256)
  2. Only the hash is sent to OTRUST
  3. OTRUST aggregates hashes and creates a Merkle tree
  4. The Merkle root is anchored to Bitcoin via OpenTimestamps
  5. You receive a .ots receipt file for independent verification
Privacy: Your files never leave your device. Only the cryptographic hash is transmitted.

Use Cases

  • Research data – Prove when experiments were conducted
  • Creative works – Establish priority for ideas and designs
  • Legal documents – Prove document existence at a specific time
  • Code commits – Timestamp important software releases
  • Audit trails – Create immutable logs of business events

Receipt history (browser-local)

Your receipt list is stored in localStorage in the web tool and in chrome.storage.local in the extension. It is not uploaded to the server and cannot be listed by public key.

  • History tab – Shows timestamps created in this browser only
  • Encrypted backup – Keys tab → export/import password-protected JSON (keys + receipts + sign history)
  • Share proof – Download a share card (PNG) or copy verification text after creating a timestamp
  • Bitcoin toast – History tab polls pending receipts and notifies when Bitcoin confirms
API note: GET /receipts/:pubkey returns 410 Gone. Persist receipts client-side when you create them, or use individual /proof/:id URLs for verification.

Technical Details

Commitment ID Packages

The current public beta uses SHA-256 field commitments and ID packages instead of server-side Circom/SNARK dependencies. ID packages are designed to reveal the claim and verification metadata without exposing the raw input.

  • Age - verifies an age threshold without storing the birthdate
  • Income - verifies a threshold without storing the exact income
  • Membership - verifies membership metadata without exposing the raw member identifier

Verification

Stored ID packages can be checked through the public verification endpoints. Advanced browser-generated IDs can still be submitted, but the backend no longer depends on a vulnerable server-side SNARK toolchain.

Auth

Use hosted ID verification for challenge-bound partner Auth.

Authentication flow

OTRUST Auth lets users authenticate with an ID package without sharing raw identity documents with the partner.

1
User starts OTRUST Auth

Your app creates an authentication challenge.

2
User opens the hosted Auth URL

The user is redirected to OTRUST to present or create an ID.

3
ID package verified

OTRUST verifies ID ownership and challenge binding without exposing underlying identity documents.

4
Callback with token and state

Your app receives a signed token and verifies it server-side before creating a session.

SDK Integration

// Create hosted Auth challenge
import { auth } from '@otrust/sdk'

const state = auth.generateState()
const challenge = await auth.createChallenge({
  clientId: 'my-app',
  redirectUri: 'https://yourapp.com/auth/callback',
  scope: ['identity'],
  state
})

// Redirect user to OTRUST
window.location.href = challenge.loginUrl

Verify Callback

// In your callback handler
const result = await auth.verify(token)

if (result.valid) {
  console.log(result.proofId) // id_xxx for audit
}

  2. Extract to a folder
  3. Chrome: chrome://extensions → Developer mode → Load unpacked

Usage

Click the OTRUST icon to:

  • Verify – Open verify flow with the current page hash
  • History – List receipts saved in this extension profile only (not synced to otrust.eu)
  • Email opt-in – Optional Bitcoin confirmation email

Extension history is separate from the web tool — each browser profile keeps its own list.

CLI Tool

Command-line interface for automation and scripting.

Installation

# Clone and install
git clone https://github.com/otrust-eu/opensource.git
cd opensource
npm install
npm link  # Makes 'otrust' command available

Commands

# Generate and save keypair
otrust keygen --save

# Timestamp a file or raw hash
otrust claim document.pdf
otrust claim a1b2c3...<64 hex>

# Verify a file or hash
otrust verify document.pdf

# Batch timestamp (max 100 files, one PoW)
otrust bulk ./docs/ --recursive

# Batch verify hashes
otrust verify-bulk ./docs/ -r --json

# Local CLI history (~/.otrust/history.json)
otrust history
otrust history --json

# Export IPFS-ready proof bundle
otrust ipfs-export ot_abc123xyz --out proof-bundle.json

Run otrust help for the full list. Email and webhook notifications are set via the API (notify_email, notify_webhook on POST /claim), not CLI flags.

Bitcoin confirmation webhook

Pass notify_webhook (HTTPS only) and optional notify_webhook_secret on POST /claim, POST /claim/bulk, or POST /claim/simple. When OpenTimestamps confirms on Bitcoin, OTRUST POSTs once and deletes the webhook URL.

{
  "event": "bitcoin_confirmed",
  "receipt_id": "ot_abc123",
  "hash": "sha256-hex",
  "block_height": 912345,
  "proof_url": "https://www.otrust.eu/proof/ot_abc123",
  "ots_url": "https://www.otrust.eu/proof/ot_abc123?format=ots",
  "confirmed_at": "2026-07-04T14:00:00.000Z"
}

If notify_webhook_secret is set, verify header X-OTRUST-Signature: sha256=<hmac> (HMAC-SHA256 of the raw JSON body).

Alternatively use notify_email for a one-time confirmation email (deleted after send).
