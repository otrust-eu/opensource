Web Tools

Three browser-first tools. No installation required; the server stores only the cryptographic metadata needed for verification.

Timestamp Tool

Prove when any file existed using Bitcoin blockchain anchoring.

1
Drop your file

Drag any file into the upload area. The file is hashed locally – nothing is uploaded.

2
Click "Timestamp"

Only the SHA-256 hash is sent to the server.

3
Save your receipt

Download the receipt JSON with your claim ID.

4
Wait for blockchain confirmation

OpenTimestamps anchors to Bitcoin every ~1 hour. Download your .ots proof when ready.

Verify a Timestamp

Go to the Verify tab, upload your original file and the .ots proof file. Or use any OpenTimestamps client:

pip install opentimestamps-client
ots verify document.pdf.ots

Privacy by Design

All three tools share the same privacy principles:

  • Client-side hashing – Timestamp files are hashed in your browser
  • Local-first signing – Documents can stay client-side, with encrypted temporary upload available when needed
  • Minimal data – The service stores hashes, signatures, public keys, timestamps, and proof metadata
  • Verifiable – All proofs can be independently verified without trusting OTRUST

SDK

Integrate OTRUST into your applications with official SDKs.

JavaScript / TypeScript

Quick Example

// Timestamp a file
import { timestamp, hashFile } from '@otrust/sdk'

const hash = await hashFile(file)
const result = await timestamp.create(hash)
console.log(result.id) // ot_xxx...

// Verify a timestamp

console.log(verification.status) // 'confirmed'

Timestamp

Prove when data existed using Bitcoin blockchain anchoring.

Timestamp flow

  1. Your file is hashed locally (SHA-256)
  2. Only the hash is sent to OTRUST
  3. OTRUST aggregates hashes and creates a Merkle tree
  4. The Merkle root is anchored to Bitcoin via OpenTimestamps
  5. You receive a .ots proof file for independent verification
Privacy: Your files never leave your device. Only the cryptographic hash is transmitted.

Use Cases

  • Research data – Prove when experiments were conducted
  • Creative works – Establish priority for ideas and designs
  • Legal documents – Prove document existence at a specific time
  • Code commits – Timestamp important software releases
  • Audit trails – Create immutable logs of business events

Technical Details

Commitment Proof Packages

The current public beta uses SHA-256 field commitments and proof packages instead of server-side Circom/SNARK dependencies. Proof packages are designed to reveal the claim and verification metadata without exposing the raw input.

  • Age - proves an age threshold without storing the birthdate
  • Income - proves a threshold without storing the exact income
  • Membership - proves membership metadata without exposing the raw member identifier

Verification

Stored proof packages can be checked through the public proof endpoints. Advanced browser-generated proofs can still be submitted, but the backend no longer depends on a vulnerable server-side SNARK toolchain.

Proof as Auth

Use zero-knowledge proofs for passwordless authentication.

Authentication flow

"Login with OTRUST" lets users authenticate using their identity proof without revealing personal data.

1
User clicks "Login with OTRUST"

Your app creates an authentication challenge.

2
User scans QR or opens link

Redirects to OTRUST to present their proof.

3
ZK proof is verified

OTRUST verifies the proof without seeing the underlying data.

4
Callback with verified claims

Your app receives verified attributes (e.g., "is adult", "is unique").

SDK Integration

// Create auth challenge
import { auth } from '@otrust/sdk'

const challenge = await auth.createChallenge({
  claims: ['unique', 'adult'],
  callback: 'https://yourapp.com/auth/callback'
})

// Redirect user or show QR
window.location.href = challenge.authUrl

Verify Callback

// In your callback handler
const result = await auth.verifyCallback(token)

if (result.verified) {
  console.log(result.claims) // { unique: true, adult: true }
  console.log(result.proofId) // prf_xxx for audit
}

Partners can attach an approved hosted-login theme to a client_id. The theme affects logo, colors, headline, spacing, radius, and footer copy on /auth/login?challenge=....

// Admin/dashboard save

X-Admin-Key: <admin-key>

{
  "logoUrl": "https://example.com/assets/logo.svg",
  "backgroundColor": "#F8FAFC",
  "primaryColor": "#12324A",
  "textColor": "#0F1B2D",
  "fontFamily": "Inter",
  "borderRadius": 8,
  "spacingScale": "default",
  "headline": "Sign in to Acme",
  "subhead": "Secure verification with OTRUST Proof",
  "footerText": "Acme uses OTRUST as identity provider",
  "autoRedirectSeconds": 3
}
Security: OTRUST disclosure, the OTRUST footer mark, and the DPO contact stay visible. The same theme follows /proof?auth_challenge=... when a user needs to create a proof first. Theme values are sanitized and all theme changes are written to the audit log.

CLI Tool

Command-line interface for automation and scripting.

Installation

# Clone and install
git clone https://github.com/otrust-eu/opensource.git
cd opensource
npm install
npm link  # Makes 'otrust' command available

Timestamp Commands

# Timestamp a file
otrust claim document.pdf

# Timestamp with email receipt
otrust claim document.pdf --email you@example.com

# Timestamp multiple files
otrust claim *.pdf

# Check status
otrust status ot_abc123xyz

# Download proof when ready
otrust proof ot_abc123xyz -o document.pdf.ots

# Verify a proof
otrust verify document.pdf document.pdf.ots

Signature Commands

# Create signature request
otrust sign create document.pdf --parties alice@x.com,bob@y.com

# Check signature status
otrust sign status sig_xxx

# Sign as a party
otrust sign approve sig_xxx --token xxx
